# SCL_HOT / SCL_COLD

Макросы подсказок о частоте вызова функции.

- Заголовок: `#include <scl/utility/attribute/hotcold.h>`
- Агрегатор: `#include <scl/utility/attribute.h>`

---

## SCL_HOT

Помечает функцию как часто вызываемую. Компилятор оптимизирует её более
агрессивно и размещает в горячей секции бинарного файла для улучшения
локальности инструкционного кэша. Ветви кода, ведущие к вызову функции
с `SCL_HOT`, неявно считаются likely.

Размещается перед возвращаемым типом.

### Определение наличия

| Условие | Раскрытие |
|---------|-----------|
| `__has_cpp_attribute(gnu::hot)` (GCC, Clang) | `[[gnu::hot]]` |
| `__has_attribute(hot)` (GCC, Clang старше) | `__attribute__((hot))` |
| Ни одно не выполнено | *(пусто — подсказка отсутствует)* |

### Использование

```cpp
#include <scl/utility/attribute/hotcold.h>

SCL_HOT void dispatch(const Event& e);

SCL_HOT int lookup(std::uint32_t key);
```

---

## SCL_COLD

Помечает функцию как редко вызываемую. Компилятор оптимизирует её по размеру,
а не по скорости, и размещает в холодной секции бинарного файла, отдельно от
горячего кода. Ветви, ведущие к вызову `SCL_COLD`, неявно считаются unlikely.

Размещается перед возвращаемым типом. Естественно сочетается с `SCL_NOINLINE`
и `SCL_NORETURN` для обработчиков ошибок и фатальных путей.

### Определение наличия

| Условие | Раскрытие |
|---------|-----------|
| `__has_cpp_attribute(gnu::cold)` (GCC, Clang) | `[[gnu::cold]]` |
| `__has_attribute(cold)` (GCC, Clang старше) | `__attribute__((cold))` |
| Ни одно не выполнено | *(пусто — подсказка отсутствует)* |

### Использование

```cpp
#include <scl/utility/attribute/hotcold.h>

SCL_COLD SCL_NOINLINE void report_oom(std::size_t requested);

SCL_COLD SCL_NORETURN void fatal(const char* msg);
```

## Связь с другими макросами атрибутов

### vs `SCL_FORCE_INLINE`

Это дополняющие, а не перекрывающие стратегии:

| | `SCL_FORCE_INLINE` | `SCL_HOT` |
|---|---|---|
| Что делает | Встраивает тело функции в каждое место вызова | Функция остаётся вызываемой; оптимизируется агрессивнее |
| Функция в бинарнике | Устраняется (или остаётся как запасная копия) | Присутствует; перемещается в секцию `.text.hot` |
| Влияние на I-кэш | Код размножается по всем call-site'ам | Горячие функции группируются рядом — лучше локальность |

`SCL_FORCE_INLINE` размножает код; `SCL_HOT` его консолидирует.
Сочетание `SCL_HOT` + `SCL_NOINLINE` — идиоматично: одно хорошо
оптимизированное тело без раздувания каждого места вызова.

### vs `SCL_LIKELY` / `SCL_UNLIKELY`

| Инструмент | Область действия | Механизм |
|------------|-----------------|----------|
| `SCL_HOT` / `SCL_COLD` | Вся функция | Раскладка бинарника + уровень оптимизации для всей функции |
| `SCL_LIKELY` / `SCL_UNLIKELY` | Одна ветвь (инструкция) | Подсказка о раскладке ветви в одном `if`/`switch` |
| `SCL_LIKELY_EXPR` / `SCL_UNLIKELY_EXPR` | Одна ветвь (выражение) | `__builtin_expect` в одном конкретном условии |

`SCL_COLD` неявно делает **все** ветви, ведущие к вызову этой функции,
unlikely — эффект на уровне всей функции. `SCL_UNLIKELY` / `SCL_UNLIKELY_EXPR`
действуют только на одну конкретную ветвь в месте вызова.

### Типичные комбинации

```cpp
// Горячий диспетчер — оптимизировать агрессивно, но не инлайнить,
// чтобы не раздувать каждое место вызова
SCL_HOT SCL_NOINLINE void dispatch(Event e);

// Холодный обработчик ошибок — убрать подальше от горячего кода
SCL_COLD SCL_NOINLINE SCL_NORETURN void fatal(const char* msg);

// Внутри горячей функции — точечная подсказка на уровне ветви
SCL_HOT int process(const Packet* p) {
    if (SCL_LIKELY_EXPR(p->is_valid()))
        return handle(p);
    SCL_UNLIKELY { return drop(p); }
}
```

## Примечания

- `SCL_HOT` и `SCL_COLD` взаимно исключают друг друга для одной функции.
- В MSVC аналогов нет; запасной вариант — пустая строка, код компилируется штатно.
- Каждый макрос можно переопределить до включения заголовка через
  `#define SCL_HOT` или `#define SCL_COLD`.
